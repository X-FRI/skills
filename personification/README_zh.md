# personification

`personification` 是一个写作风格类 skill，用来让回复更像“有人认真写出来的文字”，而不是标准助手模板。

它不改变任务本身做什么，改变的是回答的表达方式：减少模板腔、助手腔和机械修饰，让语气、节奏、判断和段落推进更自然。

## 这个 Skill 的作用

这个 skill 的核心作用，是把输出从“默认 AI 回答风格”往“有人在思考、有人在写作”的方向拉。

它主要会压制这类常见问题：

- 套话式开头和结尾
- 机械过渡词和流水账结构
- “作为 AI 助手”一类自我指涉
- 过度光滑、过于标准、没有作者痕迹的表达

同时，它会鼓励这些特征：

- 自然的段落推进
- 更明确的作者判断
- 更像邮件、备忘、评论或短文的书写感
- 在保留分析力度的前提下，让语言更有人味

重点不是“表演人格”，也不是“假装真人身份”。重点是让文本更好读、更少机器味，同时不牺牲信息和推理。

## 适用场景

当用户有这些需求时，适合使用这个 skill：

- 希望回复更像真人写的
- 希望降低 AI 腔或模板感
- 希望语气更自然、更有个人风格
- 希望回答像邮件、评论、说明文字或深度回复，而不是机械问答

它尤其适合这些内容类型：

- 长段落解释
- 分析型评论
- 偏编辑式或偏表达型写作
- 邮件感回复
- 有观点、有节奏的总结或讨论

## 不适用的场景

如果输出必须严格结构化、机器可读、或有固定格式约束，就不应该优先使用这个 skill。

例如：

- JSON
- 固定 schema
- 标准化工单
- 严格模板文本
- 以步骤清单为主、风格越少越好的操作说明

另外，在法律、医疗、合规等高风险场景里，这个 skill 也只能作为次级风格约束，不能压过准确性、边界表达和风险提示。

## 它具体改变什么

这个 skill 主要从四个方向改变默认输出：

1. 让段落而不是模板来驱动文本展开。
2. 去掉助手式自我呈现。
3. 保留适度的不对称和自然转折，而不是强行写得过于工整。
4. 保持推理和结论的完整性，不把“有人味”变成“更模糊”。

最后这一点很重要。这个 skill 不是纯修辞润色层，它不应该拿准确性去换氛围感。

## 自动识别输出语言

虽然 `SKILL.md` 本身是英文写的，但这个 skill 支持自动判断实际输出语言。

它的预期规则是：

1. 先看用户当前这条消息的主语言。
2. 如果当前消息太短、混杂多种语言，或判断不稳定，就看最近几条用户消息。
3. 优先根据用户文本来判断主语言，而不是根据助手文本判断。
4. 如果两种语言接近，就采用最近一次“有实际任务含义”的用户请求所用语言。
5. 如果还是不明确，就继承当前正在编辑的文档、笔记或上下文对象的主语言。
6. 如果仍然没有可靠信号，默认回退到英文。

这意味着：skill 文件可以保持英文，方便维护和检索，但实际输出仍然可以自动跟随中文、英文，或者其他用户当前正在使用的语言。

## 为什么这个 Skill 有意义

很多“让回答更像真人”的提示词最后都没有效果，因为它们只停留在一句很空的话，比如“请更自然一点”。

这个 skill 更具体。它不是抽象要求“像真人”，而是明确指出要削弱哪些典型 AI 写法：

- 空泛礼貌
- 可预测的结构模板
- 防御性自我定位
- 乍看精致、实则没有书写感的句子

然后再把这些限制转成一组更可执行的写作规则。

## 它不做什么

这个 skill 不意味着可以：

- 虚构个人经历
- 编造情绪状态
- 假装自己有现实身份和 biography
- 为了显得自然而牺牲事实准确性

这里的 “personification” 指的是文字层面的拟人化，不是身份层面的角色扮演。

## 使用后的理想效果

没有这个 skill 时，回复容易像一个训练过的标准助手模板。

有这个 skill 时，同样的内容应该更像一个真正读懂了问题、形成了判断、并且认真组织语言的人写出来的东西。

## 和 SKILL.md 的关系

`SKILL.md` 是给 agent 执行用的操作说明。

这个 README 是给人看的解释文档，用来说明这个 skill 为什么存在、适合什么场景、以及它的自动语言输出逻辑是怎么设计的。
